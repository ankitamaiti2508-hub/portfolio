IntelliVault — Internal Knowledge Agent

The Problem

Leadership teams in most organisations sit on top of enormous amounts of institutional knowledge locked in documents — quarterly financial reports, churn analyses, product roadmaps, board decks — spread across Google Drive folders, with no way to query them directly. The alternative is either manually searching through files or waiting for someone to pull the right number from the right report.

The problem isn't that the data doesn't exist. It's that there's no interface between leadership's questions and the documents that answer them.

What I Built

IntelliVault is a two-pipeline RAG (Retrieval-Augmented Generation) system that lets leadership query internal documents in plain English — and get cited, auditable answers sourced directly from those documents. No hallucinations, no guessing, no manual search.

How It Works

Documents flow in automatically

When a file is added or modified in Google Drive, an n8n pipeline triggers automatically. It extracts text, parses metadata from the filename (department, document type, time period), chunks the text into 2000-character segments with 400-character overlap, embeds each chunk using Cohere's embed-english-v3.0, and upserts to Pinecone with full metadata attached.

Queries get precise, cited answers

A leadership query ("What was Q3 churn in Finance?") gets embedded, filtered by department/time period metadata, run through Pinecone similarity search (top 10), reranked by Cohere cross-encoder (top 5), assembled into a context block, and sent to Llama 3.3 70B (via Groq) with a strict citation-only system prompt. The answer includes inline source citations and a full sources list.

Every answer is auditable

Each query and response is logged with source files, relevance scores, and a session ID. A confidence indicator (High / Medium / Low) signals when answers should be verified manually.

Key Design Decisions

Metadata-first filtering

Every chunk carries department, document type, date, and source. This enables scoped queries ("Q3 revenue, Finance only") that would otherwise retrieve noise from unrelated departments.

Two-stage retrieval

Pinecone casts a wide net (top 10 by vector similarity). Cohere Rerank re-scores using a cross-encoder that reads query + chunk together — the #1 vector result is the true best answer only ~60% of the time without reranking.

Citation-only generation

Llama 3.3 70B is instructed never to use outside knowledge — only the retrieved chunks. If the documents don't contain the answer, it says so. This prevents confident-sounding hallucinations, which are more dangerous than an honest "I don't know."

Filename-encoded metadata

A naming convention (Finance_FinancialReport_2024Q3_Name.pdf) lets n8n parse department, document type, and time period automatically — no manual tagging, no ingestion UI needed.

System Design

IntelliVault is two separate n8n pipelines with a shared Pinecone index. Documents flow in through Pipeline 1; leadership queries flow through Pipeline 2. They never interact at runtime — the index is the shared state.

Pipeline 1 — Document Ingestion

Google Drive Trigger

Fires on file create or update. Watches the entire IntelliVault/ folder tree, including sub-folders per department.

Download File → File Type Router

Downloads the file as binary data, then routes by type — PDF goes to the PDF extractor, plain text files go to the text file extractor. Both output a single text field.

Parse Metadata from Filename

JavaScript node parses Finance_FinancialReport_2024Q3_Name.pdf → department, document_type, time_period. No manual tagging needed.

Recursive Text Splitter

2000-character chunks, 400-character overlap. Tries paragraph breaks → line breaks → sentence boundaries → words to preserve natural language structure.

Process in Batches → Prepare Vector Payloads

Chunks are processed in a batch loop. Each chunk is packaged with its metadata into a vector payload ready for embedding.

Cohere Embed → Upsert to Pinecone

Each chunk embedded with Cohere embed-english-v3.0 (1024 dims) via the Cohere API. Upserted to Pinecone with full metadata: source file, department, time period, chunk index, and raw text for citation.

Pipeline 2 — Query & Answer

Webhook Trigger

Receives { query, department?, time_period? } via POST. The chat UI calls this endpoint directly.

Pinecone Vector Store + Embeddings Cohere

Query is embedded using Cohere (same model as ingestion) and passed to the Pinecone Vector Store node, which performs cosine similarity search and returns the top 10 most similar chunks.

Parse Results + Prepare Rerank

Pinecone results are parsed and formatted into the structure expected by the Cohere Rerank API — each chunk paired with the original query.

Cohere Rerank

Cross-encoder scores all 10 chunks against the query jointly, producing relevance scores that reflect direct answerability — not just topical similarity. Top 5 are kept; chunks below 0.3 are discarded.

Build Prompt → Llama 3.3 70B (Groq)

Top 5 chunks are assembled into a context block with source labels. Groq runs Llama 3.3 70B with a citation-only system prompt — the model is instructed to use only the retrieved chunks and never outside knowledge.

Format Final Response → Respond to Webhook

Answer, inline citations, and source metadata are packaged into a JSON response and returned to the caller. Every query is logged with scores, sources, and session ID.

Metadata Schema

Every vector in Pinecone carries this metadata — enabling precise filtering before similarity search runs.
